# 图像传输数据大小 + 云端延迟 — 知识卡片

## 一、链路各阶段数据量

```
① ISP 输出 NV12 (640×480)
   Y 平面:  640×480 = 307,200 bytes
   UV 平面: 640×240 = 153,600 bytes
   总计:    460,800 bytes (450KB)

② RGA NV12→RGB (640×640)
   Arena RGB: 640×640×3 = 1,228,800 bytes (1.2MB)

③ 帧传递 BGR CMA[0]/[1] (640×640)
   BGR: 640×640×3 = 1,228,800 bytes × 2 = 2,457,600 bytes (2.4MB, 双缓冲)

④ HW JPEG 中间 NV12 (448×448)
   448×448×3/2 = 301,056 bytes (294KB) — init 时一次性分配, 每帧复用

⑤ MPP JPEG 编码输出
   448×448 quality=85 → ~40-60KB (典型值)

⑥ base64 编码
   JPEG 50KB × 1.33 = ~67KB (+ JSON 头 ~3KB) → WebSocket 传输 ~70KB

⑦ LCD RGB565 (320×240)
   320×240×2 = 153,600 bytes (150KB)
```

**每帧 CMA 内存占用汇总：**

| Buffer | 大小 | 数量 | 合计 |
|--------|------|------|------|
| V4L2 ISP buffer | 450KB | ×4 | 1.8MB |
| Arena input RGB | 1.2MB | ×1 | 1.2MB |
| Arena outputs | ~0.3MB | ×3 | ~0.9MB |
| BGR 双缓冲 | 1.2MB | ×2 | 2.4MB |
| LCD RGB565 | 150KB | ×1 | 150KB |
| HW JPEG NV12 | 294KB | ×1 | 294KB |
| **总计** | | | **~6.7MB** (CMA 64MB 的 10%) |

**每帧 WebSocket 传输量：**

```
JPEG base64 ~67KB — 是原始 NV12(450KB) 的 15%, BGR(1.2MB) 的 5%
```

**数据量最大的不是 ISP 原始帧, 是 Arena RGB/BGR 三个 640×640×3=1.2MB 大块。**

## 二、云端延迟

### 服务端技术栈 (见 Server/)

```
云端请求链路:
  端侧 Opus encode → WebSocket → 服务端
    ├→ ASR: FunASR (本地 CPU) / DashScope ASR (云端)
    ├→ 意图分析: qwen-turbo / qwen-plus
    ├→ 视觉理解: qwen-vl-plus (DashScope)
    └→ TTS: DashScope 流式 TTS
       → WebSocket → 端侧 Opus decode → 播放
```

### 各阶段延迟 (端到端)

```
┌─ KWS 唤醒 ────────────────────────┐
│ NPU 推理: ~50ms (本地)             │
└───────────────────────────────────┘

┌─ 语音识别 ────────────────────────┐
│ Opus→PCM + 发送: ~50ms             │
│ FunASR 本地: 200-500ms             │ → 0.3-0.6s
│ DashScope ASR: 500ms-1s            │
└───────────────────────────────────┘

┌─ 意图分析 ────────────────────────┐
│ qwen-turbo API: 0.3-0.8s          │ → 0.3-1s
└───────────────────────────────────┘

┌─ 视觉理解 ────────────────────────┐
│ qwen-vl-plus: 1-3s (瓶颈)         │ → 1-3s
│ 图片编码+握手: 50ms 可忽略         │
│ 图片 70KB 传输占比 < 2%             │
└───────────────────────────────────┘

┌─ TTS 合成 ────────────────────────┐
│ DashScope 流式: 0.5-1.5s          │ → 0.5-1.5s
└───────────────────────────────────┘

════════════════════════════════════════════
端到端延迟 (从用户问完到开始播报):
  纯对话 (无图片): 1.5-3s
  带图片 (主动/被动视觉): 2.5-6s
  瓶颈: VLM 视觉推理占 50%+
════════════════════════════════════════════
```

### 换模型对比

| 模型 | 图片延迟 | 质量 | 中文 | 调用 |
|------|---------|------|------|------|
| `qwen-vl-plus` (当前) | 1-3s | 中 | 中 | DashScope |
| `qwen-vl-max` | 1.5-2.5s | 好 | 最好 | DashScope (一行改) |
| `Step-2V` (阶跃) | 0.5-1s | 好 | 好 | 新 API |
| `GLM-4V-Plus` (智谱) | 0.8-1.5s | 好 | 好 | 新 API |
| `Doubao-vision-pro` (字节) | 0.4-1s | 好 | 好 | 火山引擎 |

### 降延迟方向

降延迟的正确方向不是换模型，是改服务端逻辑：
- 流式 TTS 先播缓冲语 ("让我看看...")
- VLM 异步处理后追加播报
- 用户感知延迟从 6s 降到 ~3s
